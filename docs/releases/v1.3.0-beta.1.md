# v1.3.0-beta.1 - SecureInfra Monthly Reporting, Control Mapping, and Release Integrity

Release target date: 2026-07-01

## Summary

Version `v1.3.0-beta.1` expands SecureInfra from a one-time defensive
assessment toolkit into a more repeatable public release and monthly improvement
reporting engine.

This beta release remains public, defensive, audit-only, deterministic, and
local/offline where possible. It does not include customer data, private
commercial reporting workflows, private prompts, pricing, or branding.

## Highlights

* Improved AD/account evidence classification so account-related findings are
  easier to group and review without adding customer-specific assumptions.
* Improved missing-value handling so absent, unknown, and unavailable evidence
  can be represented more consistently in public defensive reports.
* Added release bundle integrity metadata, including `RELEASE-MANIFEST.json`
  and `SHA256SUMS.txt` support for public release artifacts.
* Added deterministic control mapping under normalized report metadata using
  broad public-safe references such as CIS Controls IG1, NIST CSF 2.0, and BSI
  SMB Security Guidance.
* Added deterministic monthly KPI trend summaries from existing normalized
  reports, including baseline summaries and month-over-month comparison support.
* Added backup readiness audit coverage for Windows and Linux backup evidence
  metadata without reading backup contents or performing restore actions.

## SecureInfra AI Reporting

This release adds broader public reporting support while keeping output
deterministic and local:

* Control references are stored under report metadata rather than changing the
  normalized finding contract.
* Monthly KPI summaries focus on current findings, new findings, persistent
  findings, resolved findings, evidence gaps, and recommended next-month focus.
* Risk reduction scoring is deterministic and explainable, and is not a formal
  risk score, compliance score, or audit attestation.
* Report language remains factual and non-alarmist.

## Release Integrity

The public release bundle workflow now supports:

* Selected public scripts, docs, schemas, examples, and SecureInfra_AI files.
* `RELEASE-MANIFEST.json` with file path, size, SHA256 hash, release version,
  and generation timestamp.
* `SHA256SUMS.txt` for local artifact verification.
* Exclusions for repository metadata, reports, customer data, private files,
  local configuration, generated outputs, raw bundles, secrets, credentials,
  temporary files, and archive files.

Optional signing remains documentation-only and operator-controlled. The public
repository does not include signing keys, certificates, tokens, credentials, or
private key material.

## Backup Readiness Coverage

Backup readiness support is audit-only and metadata-oriented. It can help flag
stale backup evidence, missing expected backup paths, restore-test gaps,
monitoring gaps, and unverified backup service presence.

The collectors do not read backup contents, perform restore operations, or
include customer backup data in public samples.

## Safety Boundaries

This repository remains the public defensive infrastructure security engine.

* Defensive and audit-only by default.
* No exploit code, credential theft, password spraying, Kerberos ticket
  requests, persistence, evasion, destructive automation, or offensive
  automation.
* No customer data, real customer domains, real user names, private report
  templates, private AI prompts, pricing, or private commercial workflows.
* Public examples remain fictional.
* No compliance certification, official audit status, or attestation is claimed.

## Known Limitations

* This is a beta release.
* Lab validation is recommended before production use.
* Microsoft 365 and Entra ID evidence collection and analysis are not included
  in this release.
* Patch posture reporting is planned but not included in this release.
* Some schemas and report views may continue to evolve during beta.
* Monthly KPI summaries depend on the quality and consistency of available
  normalized report evidence.
* Control mappings are broad defensive references and are not compliance
  certification, official audit evidence, or attestation.

## Validation

Before publishing or tagging this beta release, run the repository checks that
are practical for the target environment, for example:

```bash
python -m unittest discover -s tests -p "test_*.py"
```

If shell-based tests are run on Windows, use a Git Bash/MSYS environment where
the relevant shell scripts can execute consistently.
